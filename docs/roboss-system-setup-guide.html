ROBOSS Setup Guide

คู่มือตั้งค่าระบบ ROBOSS สำหรับลูกค้าและแฟรนไชส์

เอกสารฉบับนี้ใช้สำหรับตั้งค่าระบบตั้งแต่โครงสร้างพื้นฐาน, backend, customer portal, admin dashboard ไปจนถึงการตั้งค่า payment ในหลังบ้าน โดยอ้างอิงจาก flow และ env ที่ใช้งานอยู่จริงในโปรเจกต์ปัจจุบัน เหมาะสำหรับทีมลูกค้า, ทีม implement และแฟรนไชส์สาขา

ขอบเขตคู่มือ ระบบลูกค้า, หลังบ้าน, backend, database, LINE, Clerk, payment และ checklist ก่อน go-live
โฟกัสสำคัญ ให้แฟรนไชส์สามารถ setup payment ปลายทางของตัวเองได้ โดย HQ ยังควบคุม governance ได้
ข้อควรจำ ระบบจะปล่อยให้เริ่มล้างได้ต่อเมื่อ payment ถูกยืนยันสำเร็จแล้วเท่านั้น

สารบัญ

1. ภาพรวมระบบ 2. สิ่งที่ต้องเตรียม 3. การตั้งค่า Backend 4. การตั้งค่า Frontend และ Admin 5. การตั้งค่า Payment 6. Flow การชำระเงิน 7. สิทธิ์ HQ และ Franchise 8. การทดสอบก่อนเปิดใช้งาน 9. Checklist ก่อน Go-Live 10. Troubleshooting
System Overview

1. ภาพรวมระบบ

สถาปัตยกรรมของ ROBOSS แยกส่วนชัดเจนเพื่อให้ scale หลายสาขาและหลายแฟรนไชส์ได้ง่ายขึ้น

Customer Portal

หน้าลูกค้าใช้สำหรับเข้าใช้งานผ่านลิงก์หน้าเว็บหรือ LINE Rich Menu แล้วเชื่อมต่อกับ backend API และ WebSocket

Admin Dashboard

หน้าหลังบ้านใช้ให้ HQ และแฟรนไชส์บริหารสาขา, payment config, coupon, package, session และนโยบายส่วนกลาง

Backend + Machine

backend ทำหน้าที่เป็น source of truth ของ session, payment, machine event, MQTT command และ audit log

LINE Rich Menu / Web Entry
        ↓
Customer Portal
        ↓
ROBOSS Backend API + WebSocket
        ↓
PostgreSQL / Payment Provider / MQTT / Machine Events
        ↓
ESP32 / PLC / เครื่องล้าง
หลักการสำคัญถึงแฟรนไชส์จะรับเงินเข้าบัญชีของตัวเองได้ แต่การตัดสินว่า session ไหนจ่ายสำเร็จและพร้อมล้างหรือยัง ต้องเกิดจาก backend กลางเสมอ เพื่อให้ audit, machine safety และการกระทบยอดทำงานตรงกันทุกฝ่าย
Preparation

2. สิ่งที่ต้องเตรียมก่อนเริ่มตั้งค่า

แนะนำให้ล็อกข้อมูลต่อไปนี้ให้ครบก่อนเริ่ม setup จริง เพื่อลดการแก้ env หลายรอบ

Infrastructure

  • PostgreSQL พร้อมค่า DATABASE_URL
  • Backend hosting พร้อม public URL เช่น https://api.roboss.co.th
  • Customer portal URL และ admin URL จริง
  • MQTT broker URL และ credential สำหรับเครื่องล้าง

Identity และ Integration

  • LINE Login channel
  • LINE Official Account / Rich Menu
  • Clerk key ถ้าจะใช้ customer auth แบบ Clerk
  • Payment provider contract และ webhook secret
รายการต้องมีเมื่อไรหมายเหตุ
โดเมน backend / customer / adminก่อน deploy stagingใช้ผูก CORS_ORIGIN, callback URL และ Rich Menu entry
JWT / machine / payment secretsก่อนเปิด productionควรเป็นค่าใหม่ยาวอย่างน้อย 32 ตัวอักษร
Payment provider endpointก่อนรับเงินจริงproduction ห้ามใช้ scaffold adapter แบบหลอก
PromptPay ID / ชื่อรับเงินของแต่ละสาขาก่อนเปิดสาขาใช้งานจริงต้องกรอกในหลังบ้านหรือ seed เข้าระบบให้พร้อม
Backend Setup

3. การตั้งค่า Backend

backend เป็นส่วนที่สำคัญที่สุด เพราะเป็นตัวตรวจสิทธิ์, สร้าง session, สร้าง payment และสั่งงาน machine

01
เตรียม env productionตั้งค่า DATABASE_URL, JWT_SECRET, JWT_REFRESH_SECRET, SCAN_TOKEN_SECRET, BRANCH_PAYMENT_CREDENTIAL_SECRET, CORS_ORIGIN, MACHINE_EVENT_SECRET และตัวแปร payment/webhook ให้ครบ
02
ตรวจความพร้อม envรันคำสั่งตรวจสอบก่อนเสมอ เพื่อจับค่าที่ placeholder, weak secret หรือ config ที่ production ไม่อนุญาต
03
migrate database และ buildใช้ migration จริงกับ production database และทดสอบ build / smoke test ก่อนเปิด endpoint ให้ลูกค้าเข้า
cd backend
npm install
npm run verify:env
npm run verify:production
npx prisma migrate deploy --schema prisma/schema.prisma
npm test
npm run build
ENV สำคัญหน้าที่ข้อกำหนด
DATABASE_URLเชื่อม PostgreSQLต้องเป็น production DB จริง
CORS_ORIGINกำหนด frontend/admin ที่อนุญาตห้ามมี localhost หรือ * ใน production
JWT_SECRET, JWT_REFRESH_SECRETออก token ให้ระบบต้องยาวและไม่ใช้ค่า default
SCAN_TOKEN_SECRETใช้กับ machine scan tokenสำคัญต่อความปลอดภัยของ flow หน้าเครื่อง
BRANCH_PAYMENT_CREDENTIAL_SECRETเข้ารหัส credential payment ของสาขาห้ามใช้ค่าอ่อนหรือซ้ำกับ secret ตัวอื่น
MACHINE_EVENT_SECRETยืนยัน event จากฝั่งเครื่องproduction ต้องมีเสมอ
Guardrail ของระบบproduction ถูกออกแบบให้ reject config ที่ไม่ปลอดภัย เช่น PAYMENT_ALLOW_MANUAL_CONFIRM=true, ไม่มี PAYMENT_PROVIDER_WEBHOOK_SECRET หรือยังใช้ provider ชนิด scaffold อยู่
Frontend Setup

4. การตั้งค่า Customer Portal และ Admin Dashboard

ทั้ง 2 ส่วนแยก deploy กันได้ แต่ต้องชี้ไป backend ตัวเดียวกันและอยู่ใน CORS ที่ backend อนุญาต

Customer Portal

  • VITE_API_URL ชี้ไป https://api...
  • VITE_WS_URL ชี้ไป wss://api.../ws ถ้ามี
  • VITE_CUSTOMER_PORTAL_URL เป็น URL หน้า portal จริง
  • VITE_LINE_RICH_MENU_ENTRY_URL ใช้เป็น entry จาก LINE
  • ถ้าใช้ Clerk ให้ใส่ VITE_CLERK_PUBLISHABLE_KEY

Admin Dashboard

  • VITE_API_URL ชี้ไป backend URL หลัก
  • ต้องอยู่ใน CORS_ORIGIN เช่นเดียวกับ customer portal
  • ใช้สำหรับจัดการ branch, package, coupon, payment config และ session
# customer portal
VITE_API_URL=https://api.roboss.co.th/api
VITE_WS_URL=wss://api.roboss.co.th/ws
VITE_CUSTOMER_PORTAL_URL=https://portal.roboss.co.th
VITE_LINE_RICH_MENU_ENTRY_URL=https://portal.roboss.co.th
VITE_CLERK_PUBLISHABLE_KEY=pk_live_xxx

# admin dashboard
VITE_API_URL=https://api.roboss.co.th
Payment Setup

5. การตั้งค่า Payment ในระบบ

ส่วนนี้ใช้สำหรับ HQ และแฟรนไชส์ในการกำหนดปลายทางรับเงินของแต่ละสาขา และควบคุมความพร้อมก่อนเปิดใช้งาน

Mode ที่รองรับ

  • hq_managed HQ ดูแลปลายทางรับเงิน
  • branch_managed สาขาหรือแฟรนไชส์ดูแลเอง
  • manual_promptpay ใช้ PromptPay แบบ manual ก่อน

Provider ที่รองรับในข้อมูล config

  • promptpay_manual
  • opn
  • stripe
  • bank_qr
  • custom

Governance Status

  • draft ยังไม่พร้อม
  • pending_review รอ HQ ตรวจ
  • approved ผ่านแล้ว
  • rejected ต้องแก้ไข
1
เลือกสาขาที่จะตั้งค่าเข้าเมนูตั้งค่าการชำระเงินในหลังบ้าน แล้วเลือกสาขาที่ต้องการให้รับเงินจริง
2
กำหนด mode และ providerเลือกว่าจะให้ HQ ดูแลหรือสาขาดูแลเอง จากนั้นเลือกชนิด provider ที่จะใช้รับเงิน
3
กรอกข้อมูลปลายทางรับเงินอย่างน้อยควรมี promptpay_id และ promptpay_name หรือ credential ตาม provider จริง เช่น API key, merchant id, webhook secret
4
ตั้งค่า capability ให้ตรงความสามารถจริงระบบต้องรู้ว่า provider นี้รองรับ webhook, polling, dynamic QR, reference binding หรือ refund หรือไม่
5
เปิดใช้งานและรอ HQ approve ถ้ามี governanceถ้าระบบใช้งานแบบ HQ คุมกลาง สาขาต้องส่ง config ขึ้น review ก่อนเปิดจริง
Field / Credentialจำเป็นเมื่อไรคำอธิบาย
promptpay_idแทบทุก flow ที่เป็น PromptPayเลข PromptPay ปลายทางที่สาขาจะรับเงินจริง
promptpay_nameควรมีเสมอชื่อแสดงผู้รับเงินใน QR context และเอกสารอ้างอิง
api_key / merchant_idเมื่อใช้ provider แบบ APIใช้ยิง create payment และ verify payment
webhook_secretเมื่อ provider ส่ง webhook กลับใช้ตรวจความถูกต้องของ callback
supportsReferenceBindingจำเป็นมากถ้าไม่มี ระบบจะไม่ถือว่า config พร้อมใช้งานจริง
เงื่อนไข readiness ของ payment configconfig จะถือว่า “พร้อมใช้งาน” ก็ต่อเมื่อมี config ของสาขา, มี PromptPay ID, มีชื่อรับเงิน, เปิดใช้งานไว้ และรองรับ supportsReferenceBinding
# ตัวอย่าง env ฝั่ง backend สำหรับ live payment
PAYMENT_PROVIDER_NAME=replace-with-real-provider-adapter
PAYMENT_PROVIDER_CREATE_URL=https://payments.example.com/v1/payments
PAYMENT_PROVIDER_VERIFY_URL=https://payments.example.com/v1/payments/:providerRef
PAYMENT_PROVIDER_VERIFY_METHOD=GET
PAYMENT_PROVIDER_API_KEY=replace-with-provider-api-key
PAYMENT_PROVIDER_CALLBACK_URL=https://api.roboss.co.th/api/payments/webhook/provider
PAYMENT_PROVIDER_WEBHOOK_SECRET=replace-with-unique-webhook-secret
PAYMENT_PROVIDER_SIGNATURE_SECRET=
PAYMENT_ALLOW_MANUAL_CONFIRM=false
Payment Flow

6. Flow การชำระเงินของระบบ

ใช้ส่วนนี้อธิบายให้ลูกค้าเข้าใจว่าระบบไม่ปล่อยให้เครื่องเริ่มทำงานจากการสร้าง QR อย่างเดียว แต่ต้องยืนยันว่าเงินเข้าแล้วจริง

1
ลูกค้าสแกน QR หน้าเครื่องและสร้าง sessionระบบผูก session เข้ากับ branch, machine และ scan context ของรายการนั้น
2
backend สร้าง paymentถ้า session อยู่ในสถานะรอชำระ ระบบจะเลือก payment config ของสาขา, สร้าง reference, สร้าง QR payload และบันทึก payment เป็นสถานะ pending
3
payment มีอายุจำกัดpayment ที่รออยู่มี expiry ประมาณ 5 นาที ถ้าไม่จ่ายทันจะกลายเป็น expired
4
ระบบรอยืนยันการชำระเงินการยืนยันทำได้จาก webhook ของ provider, การ verify/reconcile จากหลังบ้าน หรือ manual confirm เฉพาะกรณีพิเศษที่ไม่ได้เปิดใน production
5
เมื่อจ่ายสำเร็จ session จะกลายเป็น ready_to_washถ้าสถานะ payment กลายเป็น confirmed ระบบจะเปลี่ยน session เป็น ready_to_wash และจองเครื่องไว้ให้รายการนั้น
6
ถ้าจ่ายไม่ผ่านหรือหมดเวลาsession จะกลับไปฝั่ง payment_failed หรือสถานะที่เกี่ยวข้อง และระบบจะไม่ปล่อยให้เริ่มล้าง
Provider StatusMapped เป็นผลกับระบบ
success, paid, succeededconfirmedพร้อมล้าง
failed, declined, errorfailedกลับไปแก้รายการชำระเงิน
expired, timeoutexpiredต้องสร้าง payment ใหม่
cancelledcancelledยกเลิกรายการ
refundedrefundedคืนเงินและ update session ตามเงื่อนไข
Roles & Governance

7. สิทธิ์ HQ และ Franchise

เพื่อให้แฟรนไชส์ดูแลปลายทางรับเงินของตัวเองได้ แต่ HQ ยังควบคุมความปลอดภัยและมาตรฐานกลางได้

HQ Admin

  • ดูและจัดการ payment config ได้ทุกสาขา
  • lock config ไม่ให้สาขาแก้เองได้
  • approve / reject config ของสาขา
  • ดู governance overview และ audit log
  • ตรวจ readiness ของ payment config ทั้งระบบ

Franchise / Branch Admin

  • แก้ได้เฉพาะสาขาที่ตัวเองมีสิทธิ์
  • กรอก PromptPay ID, ชื่อรับเงิน และ provider credential ของตัวเอง
  • ส่ง config เข้าสู่สถานะ review
  • เปิด/ปิด config ได้เมื่อ HQ ไม่ lock ไว้

8. การทดสอบก่อนเปิดใช้งาน

แนะนำให้ทดสอบจากต้นทางถึงปลายทาง ไม่ใช่เช็กแค่หน้าเว็บโหลดขึ้นได้

Backend / Infra

  • เช็ก /health และ /ready ต้องตอบ 200
  • ล็อกอินแอดมินได้จริง
  • backend เชื่อม DB ได้
  • MQTT รับ event และส่ง command ได้

Payment Flow

  • สร้าง session ได้จากหน้าเครื่องหรือ customer flow
  • ระบบสร้าง payment และ reference ถูกสาขา
  • webhook หรือ verify เปลี่ยน payment เป็น confirmed ได้
  • session เปลี่ยนเป็น ready_to_wash หลังยืนยันเท่านั้น

Go-Live

9. Checklist ก่อนเปิดใช้งานจริง

ใช้รายการนี้เป็นตัวเช็กสุดท้ายก่อนเปิดรับลูกค้าจริง

ต้องผ่านทั้งหมด

  • พร้อม backend health / ready ผ่าน
  • พร้อม admin และ customer portal ใช้งานได้จากโดเมนจริง
  • พร้อม payment webhook secret และ signature config ถูกต้อง
  • พร้อม แต่ละสาขามี payment config ที่ active และ ready
  • พร้อม machine event secret และ MQTT ทำงานจริง

ห้ามเปิดใช้งานถ้ายังมีข้อใดข้อหนึ่ง

  • ห้าม ยังใช้ provider scaffold อยู่
  • ห้าม เปิด PAYMENT_ALLOW_MANUAL_CONFIRM=true ใน production
  • ห้าม ไม่มี webhook secret
  • ห้าม CORS ยังชี้ localhost
  • ห้าม promptpay_id ของสาขายังไม่ครบหรือไม่ผ่าน HQ review
Troubleshooting

10. ปัญหาที่พบบ่อยและแนวทางแก้

ใช้ส่วนนี้เป็น quick guide เวลา setup แล้ว flow ยังไม่สมบูรณ์

อาการสาเหตุที่เป็นไปได้แนวทางแก้
สร้าง payment ไม่ได้สาขายังไม่มี payment config ที่ active หรือไม่มี reference bindingเข้าเมนูตั้งค่าการชำระเงิน แล้วเช็ก promptpay_id, promptpay_name และ capability
จ่ายแล้วแต่ session ไม่พร้อมล้างwebhook ไม่เข้า, verify ไม่ผ่าน หรือ amount mismatchเช็ก webhook secret, payload mapping, verify URL และ audit/payment attempt
หน้าเว็บเข้าได้แต่เรียก API ไม่ผ่านCORS_ORIGIN ไม่รวมโดเมนที่ deploy จริงอัปเดต CORS_ORIGIN ทั้ง customer portal และ admin domain
เครื่องไม่เริ่มแม้ payment confirmed แล้วmachine ไม่พร้อม, MQTT ไม่ถึง หรือ session ไม่อยู่สถานะ ready_to_washเช็ก machine status, MQTT topic, machine event secret และ start wash guard
เอกสารนี้จัดทำจากโครงสร้างและ flow ที่มีอยู่ในโปรเจกต์ ROBOSS ปัจจุบัน หากต้องการเวอร์ชันสำหรับลูกค้าแบบใส่โลโก้บริษัท, ปุ่มพิมพ์ PDF, หรือแยกฉบับ “คู่มือแฟรนไชส์” กับ “คู่มือทีมติดตั้ง” สามารถต่อยอดจากไฟล์นี้ได้ทันที
